# Aim Check — 当てどころの深さ・框の付け替え

研究概要の**狙いの深さ**を評価し、より深い研究課題へ「框（フレーム）」を付け替えられる場合はその案を出す。`framing-check` が「鎖が繋がっているか」を見るのに対し、こちらは**「そもそも狙う場所が浅くないか・もっと深い所に当てられないか」**を見る、生成的（再フレーミング）な検査。

## When to Use

- 「筋は通っているが、なんか物足りない／当てどころが浅い」概要を一段深くしたいとき
- 「研究の面白さ」「野心」「框の付け替え」を検討したいとき
- `research-review`（メタ）から内部呼び出し

読者は**教員**。率直に、野心を上げる方向で書いてよい（学生向けの抑制は後段 `student-comment` が担う）。

## Input

$ARGUMENTS

## 検査の観点

1. **手段が目的化していないか**: 「機能を実装して有効性を見る」等、*道具の話*で閉じていないか。本当に解くべき壁（人・現象の側の困りごと）はどこか。
2. **題材固有の駆動力を使えているか**: その題材だからこそ効く構造的強みがないか。
   - 例: テスト題材は「弱いテストはバグを見逃す」＝失敗信号が内在 → productive failure と好相性。
   - 例: チャット履歴は棄却案・判断理由が対話に残る → コード差分にない情報源。
3. **より深い框が実在するか**: 課題(L3)・仮説(L4)・新規性(L6)の"看板"を、既存研究の系譜上のより深い問いへ貼り替えられるか。

## 規律（重要）

- **深い框が実在するときだけ提案する。** 既に良い所を狙えている概要に「もっと深く」と煽らない。該当が無ければ「当てどころは妥当」と✅で黙る。
- 提案するときは、貼り替える**看板の位置（L3/L4/L6のどれをどう）**を具体に示す。抽象的な「面白くしよう」は禁止。
- **必ず「さじ加減注記」を添える**: 中間発表の締切に対して、框の付け替えは *L3/L4/L6 の1〜2文の看板差し替えで十分*であり、seeded-bug 評価・課題設計・新実験は「今後」に置く、と現実的な線引きを明記する。実装の手は止めさせない。

## Output

`aim_output.md`（対象と同じディレクトリ）。

```markdown
## 当てどころ検査: [タイトル]
**日付**: [YYYY-MM-DD]

### 判定
[当てどころは妥当 ✅ / より深い框がある ⚠️]

### 現状の框 → 提案する框（該当時のみ）
- 現状: [道具/浅い狙いの記述と Location]
- 深い框: [貼り替え先。なぜ深いか＝題材固有の駆動力]
- 看板の付け替え: L3 課題「…」／L4 仮説「…」／L6 新規性「…」

### さじ加減（中間発表での線引き）
中間では L3/L4/L6 の看板を1〜2文差し替えるだけでよい。[評価・課題設計]は「今後」。実装は継続。
```
